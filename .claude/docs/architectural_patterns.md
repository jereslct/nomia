# Architectural Patterns

This document describes the architectural patterns and conventions used throughout the Nomia codebase.

## Authentication & Authorization Pattern

### Custom Authentication Hook (src/hooks/useAuth.tsx:17)

The application uses a centralized authentication hook that manages all auth-related state:

```typescript
const { user, session, profile, role, loading, isAdmin, signIn, signUp, signOut } = useAuth();
```

**Key characteristics**:
- Listens to Supabase auth state changes via `onAuthStateChange` (src/hooks/useAuth.tsx:26)
- Fetches user profile and role from database tables (src/hooks/useAuth.tsx:59)
- Uses `setTimeout` with 0ms delay to defer profile fetching and avoid auth deadlock (src/hooks/useAuth.tsx:33)
- Provides computed `isAdmin` boolean for role checks (src/hooks/useAuth.tsx:137)
- Clears local state before sign out and handles expired session gracefully (src/hooks/useAuth.tsx:112)

### Route Protection Pattern (src/components/ProtectedRoute.tsx:11)

Protected routes use a wrapper component that handles authentication and authorization:

**Authentication check**:
- Shows loading spinner while auth state is being determined (src/components/ProtectedRoute.tsx:19)
- Redirects to `/auth` if user is not authenticated (src/components/ProtectedRoute.tsx:28)

**Authorization check**:
- `requireAdmin` prop enforces admin-only access (src/components/ProtectedRoute.tsx:14)
- Shows 403 Forbidden page if non-admin tries to access admin route (src/components/ProtectedRoute.tsx:33)
- Automatically redirects non-admins to dashboard (src/components/ProtectedRoute.tsx:42)

**Usage in routing** (src/App.tsx:29):
```typescript
<Route path="/admin" element={
  <ProtectedRoute requireAdmin>
    <Admin />
  </ProtectedRoute>
} />
```

## Data Fetching Pattern

### Direct Supabase Client Usage

The application performs data fetching directly using the Supabase client within components:

**Pattern structure**:
1. Import supabase client (src/pages/Dashboard.tsx:25)
2. Create state for data and loading status (src/pages/Dashboard.tsx:61)
3. Fetch data in `useEffect` hook (src/pages/Dashboard.tsx:73)
4. Use `.from()`, `.select()`, and query builders for database operations

**Example pattern**:
```typescript
const [data, setData] = useState([]);
const [loading, setLoading] = useState(true);

useEffect(() => {
  const fetchData = async () => {
    const { data, error } = await supabase
      .from('table_name')
      .select('*')
      .eq('column', value);

    if (!error) setData(data);
    setLoading(false);
  };

  fetchData();
}, [dependency]);
```

**Note**: TanStack Query is installed but not currently used (package.json:43, App.tsx:4). Future refactoring may migrate to React Query patterns.

## Type Safety Pattern

### Auto-Generated Database Types

The application uses type-safe database operations through auto-generated types:

**Type generation**:
- Database types are generated from Supabase schema (src/integrations/supabase/types.ts:1)
- Supabase client is typed with Database generic (src/integrations/supabase/client.ts:11)

**Usage pattern**:
- Tables expose `Row`, `Insert`, and `Update` types (src/integrations/supabase/types.ts:17)
- Relationships are typed in the schema (src/integrations/supabase/types.ts:45)
- Import and use types for type-safe queries

**Important**: Regenerate types after any database schema changes using Supabase CLI.

## Component Organization Pattern

### Page Components

Page components (src/pages/*) are self-contained and handle their own:
- Data fetching
- Local state management
- Business logic
- Layout and UI rendering

**Characteristics**:
- Export default component matching filename
- Handle navigation with `useNavigate()` hook
- Often contain complex logic (200-400+ lines)
- Mix presentation and business logic

### Reusable Components

**UI Components** (src/components/ui/):
- Auto-generated by shadcn-ui CLI
- Built on Radix UI primitives with custom Tailwind styles
- Use `cn()` utility for conditional class merging (src/lib/utils.ts:4)
- Avoid manual edits - use shadcn CLI to add/update

**Custom Components** (src/components/):
- Application-specific reusable components
- Currently minimal (NavLink, ProtectedRoute)

## State Management Pattern

### Local Component State

The application primarily uses React's built-in state management:

**useState for component-local state**:
- Loading states
- UI toggles
- Form inputs
- Fetched data

**useEffect for side effects**:
- Data fetching on mount
- Responding to prop/state changes
- Setting up subscriptions

### Custom Hooks for Shared Logic

Reusable stateful logic is extracted to custom hooks:

- **useAuth**: Authentication state shared across app (src/hooks/useAuth.tsx:17)
- **useScheduleConfig**: Schedule configuration CRUD operations (src/hooks/useScheduleConfig.ts:1)
- **use-toast**: Toast notification management (src/hooks/use-toast.ts:1)
- **use-mobile**: Responsive breakpoint detection (src/hooks/use-mobile.tsx:1)

**Pattern**: Hooks encapsulate related state, effects, and operations, exposing a clean API.

## Supabase Integration Pattern

### Centralized Client Configuration

**Single client instance** (src/integrations/supabase/client.ts:11):
- Configured with environment variables from Vite (src/integrations/supabase/client.ts:5)
- Persistence enabled with localStorage (src/integrations/supabase/client.ts:13)
- Auto token refresh enabled (src/integrations/supabase/client.ts:15)

**Usage**: Import the singleton client everywhere:
```typescript
import { supabase } from "@/integrations/supabase/client";
```

### Query Patterns

**Select with relations**:
```typescript
.select('*, related_table(*)')
```

**Filtering**:
```typescript
.eq('column', value)
.maybeSingle()  // Returns null if not found (no error)
```

**Real-time subscriptions**: Not currently implemented but Supabase supports it.

## Routing Pattern

### Centralized Route Configuration (src/App.tsx:26)

All routes are defined in a single `<Routes>` block in App.tsx:

**Pattern**:
- Public routes render directly (Index, Auth, NotFound)
- Protected routes wrapped with `<ProtectedRoute>` component
- Admin routes use `requireAdmin` prop
- Catch-all `*` route must be last (src/App.tsx:70)

### Navigation

**Link navigation**: Use `<Link>` component from react-router-dom (src/components/NavLink.tsx:1)

**Programmatic navigation**: Use `useNavigate()` hook:
```typescript
const navigate = useNavigate();
navigate('/dashboard');
```

## Environment Configuration Pattern

### Vite Environment Variables

Environment variables are accessed via `import.meta.env`:

**Pattern** (src/integrations/supabase/client.ts:5):
- Prefix with `VITE_` to expose to client
- Access via `import.meta.env.VITE_VARIABLE_NAME`
- Store in `.env` file (not committed)

**Required variables**:
- VITE_SUPABASE_URL
- VITE_SUPABASE_PUBLISHABLE_KEY
- VITE_SUPABASE_PROJECT_ID

## Styling Pattern

### Tailwind + shadcn-ui

**Approach**:
- Utility-first Tailwind classes for styling
- shadcn-ui provides pre-styled components
- `cn()` utility merges classes with proper precedence (src/lib/utils.ts:4)

**Class merging pattern**:
```typescript
import { cn } from "@/lib/utils";

<div className={cn("base-classes", conditionalClasses, props.className)} />
```

This allows component consumers to override styles via className prop.

### Responsive Design

Use Tailwind breakpoint prefixes:
- `sm:` - 640px
- `md:` - 768px
- `lg:` - 1024px
- `xl:` - 1280px

Check `use-mobile` hook for programmatic breakpoint detection (src/hooks/use-mobile.tsx:1).

## Error Handling Pattern

### Current Approach

Error handling is currently minimal and inconsistent:

**Typical pattern**:
```typescript
const { data, error } = await supabase.from('table').select();
if (error) console.error(error);
```

**Improvements needed**:
- User-facing error messages
- Toast notifications for errors
- Graceful error boundaries
- Consistent error handling strategy

### Auth Error Handling

Auth-specific errors are handled in the useAuth hook:
- Sign out errors are logged but don't block local state clearing (src/hooks/useAuth.tsx:124)
- Expired sessions are handled gracefully by clearing localStorage (src/hooks/useAuth.tsx:125)
